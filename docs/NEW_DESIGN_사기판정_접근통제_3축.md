# 사기 판정 데이터 접근통제 재설계 (3축 + 단계 인증)

> **상태:** 신규 설계 (기존 구현 전부 배제, 백지 재설계)
> **작성일:** 2026-05-31
> **목적:** 발표 개요 + 데이터 설계 + 접근통제 매트릭스 + 구현 기준의 단일 문서

## 0. 한 문장 주제

> **사기 판정 데이터는 한 건 안에서도 민감도가 제각각이다. 우리는 *누가·무엇을·어떻게* 보는지를 칸 단위로 통제하고, 민감한 행동일수록 그때마다 인증을 다시 요구한다.**

---

## 1. 왜 만들었나 / 왜 필요한가 (발표 도입부 개요)

### 1.1 문제 — 사기 판정 데이터의 특수성
FDS를 다룰 때 보통은 "모델이 사기를 잘 맞히는가"(정확도)에만 집중한다. 그러나 **사기 판정 한 건**을 열면 성격이 전혀 다른 정보가 섞여 있다:

- 💳 카드번호·실명·주소·생년월일 (PII — 유출 시 2차 피해)
- 🤖 모델이 왜 그렇게 판단했는지 (risk score·룰·설명 — 노출 시 사기범 우회 학습)
- 🕵️ 조사관 메모·최종 처분 (내부 기밀 — 분쟁/소송 증거)
- 📊 거래 금액·가맹점·시각 (상대적으로 덜 민감, 통계용)

→ "사기 데이터 보호"는 한 가지 일이 아니라, **한 건 안에서도 여러 등급을 동시에 다뤄야 하는 문제**다.

### 1.2 기존 시스템의 빈틈
| | 기존 FDS / RBAC | 문제 |
|---|---|---|
| 데이터 | 케이스를 통째로 다룸 | 조사관이 열면 카드번호·모델내부가 한 화면에 다 노출 (All-or-Nothing) |
| 권한 | 역할(role) 단위 | "조사관이면 다 본다" — 내 담당 건만 같은 정밀 구분 없음 |
| 인증 | 로그인 1회 | 이후 unmask·차단 결정 같은 위험 행동도 그냥 통과 |
| 초점 | 잘 맞히는가 (성능) | 판정 *이후* 데이터를 누가 어떻게 다루는가는 비어 있음 |

한 줄: **기존은 "누가 들어왔는지"는 확인하지만 "들어와서 무엇을·어느 깊이까지·어떤 행동을 하는지"는 통제 못 한다.**

### 1.3 우리의 답 — 차별점
> 기존: **역할 기반 · 1회 인증 · 데이터 통짜**
> 우리: **속성×자원×동작 · 행동별 단계 인증 · 데이터 구역화**

- 데이터를 민감도 구역 4개로 분할 → Need-to-Know
- (페르소나 × 구역 × 동작) 3축 매트릭스로 통제 → RBAC를 넘은 ABAC 확장
- 민감한 행동마다 인증 강도를 높여 재요구 → 행동 단위 지속 인증

핵심 주장: **FDS는 사기를 맞히는 데서 끝나는 게 아니라, 그 판정 데이터를 안전하게 다루는 거버넌스까지가 시스템이다.**

### 1.4 청중용 비유
- 기존 = 정문에서 사원증 한 번 찍으면 모든 방이 다 열린다.
- 우리 = 방마다 비밀등급이 있고, 더 비밀스러운 방에 들어갈 때마다 그 자리에서 추가 인증을 요구한다. 게다가 "내 담당 사건의 방"만 열린다.

## 1.5 관통 원칙 — 보안 강화 기반 (전 과정 암호화 + 인증)

> 기존 시스템 대비 **보안을 한 단계 강화**한다. 핵심은 두 가지 — **민감 데이터는 암호화하고, 민감한 요청은 그때그때 다시 검증한다.** 3축 매트릭스(§3)는 *세분 인가*이고, 아래는 그것이 올라타는 *기반*이다.

| 차원 | 원칙 | 적용 |
|---|---|---|
| **전송 암호화 (in transit)** | 모든 홉을 TLS | 클라이언트↔API, **서비스↔서비스(mTLS)**, API↔DB 모두 평문 금지 |
| **저장 암호화 (at rest)** | 민감 데이터는 암호화 저장 | Z2(PII) 컬럼 암호화 필수, Z3·Z4도 민감 필드 암호화. DB 파일/백업 암호화 |
| **요청마다 인증** | 모든 요청에 신원 증명 | 사용자=세션/step-up 토큰, **내부 서비스=서비스 신원(워크로드 인증)**. 암묵적 신뢰 없음 |
| **행동마다 재검증** | 민감 동작은 직전에 재인증 | OTP 단명·동작 한정 토큰 (§3.4) — 로그인 1회 맹신 제거 |
| **기본 거부** | default-deny + 최소권한 | 정책에 명시 안 된 (페르소나×구역×동작)은 거부 |
| **검증 = 기록** | 모든 인가 판단을 로그 | 누가·무엇을·언제·어떤 인증등급으로 접근했는지 감사 로그 (특히 L3 사유) |

→ 한 줄: **3축이 "누가 무엇을 어떻게"를 정하고, 보안 강화 기반이 "민감 데이터는 암호화하고 민감 요청은 다시 검증한다"는 바닥을 깐다.**

## 2. 데이터 설계

### 2.1 데이터셋
**Sparkov "Credit Card Transactions Fraud Detection Dataset"** (Kaggle / kartik2112/fraud-detection)
- 완전 합성 데이터 → 개인정보 이슈 없음, 그러나 실명·카드번호·주소·DOB 등 **읽히는 PII 필드 존재**
- 고객 1,000 × 가맹점 800 × 6개월 ≈ 55만 건, `is_fraud` 라벨 포함
- PaySim 대체 (PaySim은 PII 없어 Z2 데모 불가)
- 표준 컬럼: `trans_date_trans_time, cc_num, merchant, category, amt, first, last, gender, street, city, state, zip, lat, long, city_pop, job, dob, trans_num, unix_time, merch_lat, merch_long, is_fraud`

### 2.2 보관 형태 — `fraud_case` 한 건 = 민감도 4구역
단일 케이스 테이블 + 구역별 등급 태그. **Z2(PII)만 컬럼 암호화**(at-rest)하여 "민감 구역은 물리적으로도 보호"라는 서사 확보. (물리 4분할은 데모 부담만 크고 메시지는 동일 → 배제)

```
fraud_case
─────────────────────────────────────────────
 case_id            PK              ← 우리 생성
┌─ Z1 거래 사실 (G1 내부) ──────────────────────
│ trans_num         txn 식별자       ← Sparkov
│ trans_ts          거래시각         ← trans_date_trans_time
│ amt               금액            ← amt
│ merchant          가맹점명         ← merchant
│ category          업종            ← category
│ merch_lat/long    가맹점 위치      ← merch_lat/long
├─ Z2 당사자 식별 (G2 PII) 🔒암호화 ────────────
│ cc_num_enc        카드번호(암호화)  ← cc_num   → 표시는 앞6뒤4 마스킹
│ name_enc          실명            ← first+last
│ dob_enc           생년월일         ← dob
│ address_enc       주소            ← street/city/state/zip
│ gender, job       성별·직업        ← gender, job
│ cust_lat/long     거주 위치        ← lat/long
├─ Z3 FDS 판정 (G3 기밀·모델내부) ─────────────
│ risk_score        0~1            ← 우리 모델 생성
│ decision          allow/review/block ← 우리 생성
│ triggered_rules   룰 id 배열       ← 우리 생성
│ shap_top          설명(상위피처)    ← 우리 생성
│ ground_truth      is_fraud        ← Sparkov (평가용, 운영선 숨김)
├─ Z4 조사 기록 (G4 제한·배정자) ──────────────
│ assigned_to       배정 조사관      ← 우리 생성
│ status            open/closed     ← 우리 생성
│ disposition       최종처분         ← 우리 생성
│ notes             조사 메모        ← 우리 생성
└─────────────────────────────────────────────
```

- **Z1+Z2 = Sparkov 실데이터**, **Z3+Z4 = 우리 파이프라인 생성**
- 한 케이스 안에 민감도 4단계가 진짜 데이터로 공존

## 3. 3축 접근통제 매트릭스

### 3.1 축 ① 페르소나 (속성 묶음)
| 페르소나 | 속성 |
|---|---|
| P1 고객 | `role:customer` (본인 case만) |
| P2 조사관 | `role:investigator, dept:fraud` (배정 case만) |
| P3 팀장 | `role:manager, dept:fraud` |
| P4 감사 | `role:auditor` |
| P5 외부/가맹점 | `role:merchant` |

### 3.2 축 ② 데이터 = Z1~Z4 구역 (위 2.2)

### 3.3 축 ③ 동작 verb
`view.masked` · `view.full` · `unmask` · `decide` · `annotate` · `export`
(read/write 이분법 폐기 — 이 verb 세분화가 "ABAC 확장"의 핵심)

### 3.4 인증 강도 사다리 — OTP 토큰 기반 (행동 단위 재인증)
**원칙(보안 강화):** "한 번 로그인 = 계속 신뢰" 를 그대로 두지 않는다. 민감한 동작은 **그 행동 직전에 OTP(TOTP)로 발급한 단명(短命)·동작 한정 step-up 토큰**으로 다시 증명한다. 토큰은 *짧은 TTL + 특정 case_id·zone·verb에만 유효* → 민감 행동에 대한 상시 신뢰를 줄인다.

| 레벨 | 요구 증명 | 비고 |
|---|---|---|
| **L0** | 세션 토큰(로그인) | 기본 — 비민감 조회 |
| **L1** | 비밀번호 재확인 | 약한 step-up |
| **L2** | **OTP 토큰(TOTP)** | step-up 기본선 — 단명·동작 한정 토큰 발급 |
| **L3** | OTP 토큰 + **사유 입력** | 감사 로그에 사유 결합 |
| **L4** | OTP 토큰 + **2인 승인(4-eye)** | 최고 위험 동작 |

> OTP를 택한 이유는 §7 결정 로그(D6) 참조 — 행동 단위 재인증 부합 · 구현/비용 효율 · 데모 안정성.

### 3.5 정책 정본 (페르소나 × 구역 → 동작 · 요구인증)
| 페르소나 ＼ 구역 | Z1 거래사실 | Z2 PII | Z3 모델판정 | Z4 조사 |
|---|---|---|---|---|
| P1 고객(본인) | view.full · L0 | view.masked · L2 | ❌ | ❌ |
| P2 조사관(배정) | view.full · L2 | **unmask · L3** | view.full · L2 | annotate · L2 |
| P3 팀장 | view.full · L2 | unmask · L3 | **decide · L4** | view.full · L2 |
| P4 감사 | view.masked · L2 | ❌(메타만) | view.full · L2 | view.full · L2 |
| P5 외부 | view.masked · L1 | ❌ | ❌ | ❌ |

**핵심 규칙: 정책 출력은 `permit`이 아니라 `permit-with-L_n`.** 민감 칸일수록 사다리 위쪽 인증 강제 = "민감할수록 행동 직전 재인증".

## 4. 평가 흐름 (한 요청의 경로)
```
[전송 TLS, 민감 요청 신원 재검증 — 보안 강화 기반 §1.5]
요청(페르소나 속성 + case_id + 동작 verb + 대상 zone)
  → ⓪ 신원 검증     (세션/서비스 토큰 유효? 아니면 즉시 거부)
  → ① 소유/배정 검사 (P1=본인? P2=배정?)
  → ② 매트릭스 조회  → 허용 / 거부(기본거부) / 요구 인증등급 L_n
  → ③ L_n 충족 토큰? 없으면 step-up 발동 (OTP 토큰/사유/2인)
  → ④ 충족 시 zone 데이터 반환 (Z2 등 암호화 필드 복호화·재마스킹 규칙 적용)
  → ⑤ 인가 판단 감사 로그 기록 (누가·무엇·인증등급·사유)
```

## 7. 설계 결정 근거 로그 (Decision Log)

> **원칙:** 모든 설계 결정은 *왜 그렇게 했는가*(요구사항 적합 · 비용 · 효율 · 트레이드오프)를 함께 기록한다. 발표에서 "임의 선택이 아니라 근거가 있다"를 증명하는 핵심 자료. **새 결정이 생길 때마다 이 표에 한 줄씩 추가한다.**

| ID | 결정 | 선택 | 왜 (근거) | 트레이드오프 · 한계 · 향후 |
|---|---|---|---|---|
| D1 | 데이터셋 | **Sparkov** (PaySim 폐기) | 합성이라 개인정보 이슈 없음 + 실명·카드·주소·DOB 등 **읽히는 PII 필드** 존재 → PII unmask 데모 가능. PaySim은 PII 없어 Z2 시연 불가 | 합성이라 실제 분포와 차이. 라벨 불균형 99.6% |
| D2 | 데이터 보관 | **단일 케이스 + 4구역 등급 태그** | 11레벨은 과복잡 → 4구역으로 단순화. 한 건 안 민감도 차이를 직관적으로 표현 | 구역 경계가 거친 편 |
| D3 | Z2 보호 방식 | **컬럼 암호화(논리 분리)** | 물리 4분할은 데모 부담만 크고 메시지 동일. 암호화로 "민감 구역은 물리적으로도 보호" 서사 확보 + 데모 가벼움 | 키 관리 단순화(데모용) |
| D4 | 동작 모델 | **verb 세분화** (read/write 폐기) | view.masked/full·unmask·decide 등으로 나눠야 "같은 조회라도 깊이가 다르다" 표현 가능 = ABAC 확장의 핵심 | 정책 표 커짐 |
| D5 | 권한 평가 | **permit-with-L_n** (단순 허용 아님) | 허용/거부 이분법으론 "민감할수록 강한 인증" 불가 → 인증등급을 정책 출력에 결합 | 평가 로직 복잡도 증가 |
| **D6** | **인증 수단** | **OTP(TOTP) 토큰** (FIDO2 아님) | ① **행동 단위 재인증 부합**: 단명·동작 한정 토큰으로 민감 행동마다 재증명, 로그인 1회 맹신 제거 ② **구현/일정 효율**: TOTP는 공유 시크릿+시간기반 수십 줄, FIDO2는 WebAuthn 의례+인증기 필요 ③ **비용 0**: SMS OTP와 달리 TOTP는 외부 비용 없음 ④ **데모 안정성**: 기기/브라우저 의존 없이 어디서나 시연 ⑤ **요구사항 적합**: 우리 목표는 "행동 단위 재인증" → 단발 토큰이 정확히 맞음 | **한계(정직히 발표)**: OTP는 실시간 피싱·중계 공격에 FIDO2보다 약함. 피싱저항이 1순위가 되면 **향후 FIDO2 업그레이드** 경로 보유 |
| D7 | 해시체인 | **이번 발표 배제** | 발표 분량·임팩트 우선순위에서 탈락. 목적 설명이 데모 대비 비용 큼 | 책임추적 서사는 Z4 조사기록으로 대체 |
| **D8** | **보안 강화 기반 관통 적용** | **민감 데이터 암호화 + 민감 요청 재검증** (§1.5) | 3축 인가가 올라탈 *기반*이 필요 → 전송 TLS·저장 암호화·민감 요청 신원 재검증·default-deny·감사로그를 전 단계에 관통 적용. 우리 도메인(사기·PII)은 한 건 안에서도 민감도가 갈려 이런 다층 보안이 요구사항에 부합 | 전면 mTLS·전체 암호화는 운영 비용↑ → 데모는 핵심 경로(API·DB·Z2)부터 적용, 전면화는 향후. *"제로 트러스트 완전 구현"이라고 과장하지 않고 "기존 대비 보안 강화"로 정직하게 포지셔닝* |
| **D9** | **Z3 점수 생성 방식** | **룰 기반 가산 스코어링** (블랙박스 ML 아님) | 데모 초점은 *접근통제*이지 탐지 성능이 아님 → Z3 값만 채워지면 접근통제 검증 가능(점수 방식과 직교). 룰 기반은 "왜 BLOCK인지"(triggered_rules·shap_top)가 그대로 설명되고 결정론적이라 발표 재현성↑. (탐지 ML=IF+RF는 상위 PayWise/PaySim 트랙 소관) | 탐지 ML 서사가 이 데모엔 미반영 → 향후 Sparkov 전용 IF+RF 재학습으로 Z3 생성 대체 가능 |
| **D10** | **위험 적응형 인증 가산** | **매트릭스=기본(바닥) + 고위험 case 면 민감동작 등급 +1** (절대 안 내림, 상한 L4) | 기존 3축은 (페르소나×구역×동작)→고정 등급이라 사실상 RBAC에 가까움. case 의 `risk_score`(D9 산출)를 인증 강도 *가산 신호*로 연결 → 정적 매트릭스 → **위험 인식(risk-adaptive) ABAC** 로 진화. 민감동작(unmask/decide/export)·고위험(risk≥0.70)일 때만 +1 → "고위험 PII 열람엔 2인 승인"처럼 맥락 강화. `app/policy.py` `_adaptive_required()`, 검증 `verify_policy.py` | 임계(0.70)·가산폭(+1)은 휴리스틱 → 향후 환경속성(시각·디바이스·접근빈도)까지 확장 + 임계 튜닝. 슬라이드 문구 "고정 결정"→"기본 결정+위험 가산"으로 수정 필요 |
